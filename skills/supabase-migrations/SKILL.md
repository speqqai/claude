# Supabase Migrations

Schema-only changes. Never edit applied migrations. Never run SQL by hand against dev or prod.

## Reference

- Migrations guide: https://supabase.com/docs/guides/deployment/database-migrations
- CLI reference root: https://supabase.com/docs/reference/cli/introduction
- Per-command pages used below: `migration new`, `migration list`, `migration up`, `migration repair`, `db diff`, `db push`, `db reset`, `link`.

## Commands

All commands assume `supabase` CLI on PATH (or `npx --yes supabase ...`). Wrap with Doppler when a remote DB URL or project ref is needed (see next section). Per Supabase, all `--db-url` values must be percent-encoded.

### Create
- `supabase migration new <name>` — create `supabase/migrations/<timestamp>_<name>.sql`. Timestamp is auto-generated.
- `supabase db diff -f <name>` — generate a migration file from the diff between local DB and local migrations (uses a shadow database). Inspect the file before committing.
- `supabase db diff --linked -f <name>` — diff local migrations against the linked remote project.

### Apply locally
- `supabase migration up --db-url <postgres-url>` — apply pending local migrations to a database. This repo uses this form inside the dev tools container; see `docker/db.sh` (`restore-local`).
- `supabase db reset --local` — drop the local DB and re-apply all migrations from scratch. Add `--no-seed` to skip the seed script. Use `--last <n>` or `--version <ts>` to scope.

### Push to remote
- `supabase db push --linked` — push pending local migrations to the linked Supabase project.
- `supabase db push --linked --dry-run` — print the migrations that would be applied; do not apply.
- `supabase db push --db-url <postgres-url>` — push to a non-linked project (URL must be percent-encoded).
- Optional: `--include-all` (include migrations not in remote history), `--include-roles` (`supabase/roles.sql`), `--include-seed` (seed from config). Do not use `--include-seed` against prod.

### Inspect
- `supabase migration list --linked` — show local-vs-remote migration table. Source of truth for promotion state.
- `supabase migration list --db-url <postgres-url>` — same, against an explicit URL.
- `supabase link --project-ref <ref>` — associate the working directory with a remote project. Required before `--linked` flags.

### Repair
- `supabase migration repair <version> --status applied --linked` — mark a remote migration as applied without running its SQL. Use only to reconcile drift.
- `supabase migration repair <version> --status reverted --linked` — mark a remote migration as reverted.
- Repo-specific: `node scripts/db/repair-migration-history.mjs --doppler-config <env>` (dry-run by default; add `--apply`) fixes malformed `version`/`name` rows in `supabase_migrations.schema_migrations`. See `scripts/db/repair-migration-history.mjs`.

## Secrets via Doppler

Doppler is the only sanctioned source for project refs, DB passwords, and Supabase URLs. Doppler project: `speqq`. Configs: `local`, `dev`, `prod`, `tools` (`.claude/SYSTEM.md`).

Wrap any command that reaches a remote database:

```sh
doppler run --project speqq --config <dev|prod> -- supabase <subcommand> ...
```

Or fetch a single value when constructing a URL:

```sh
doppler secrets get DATABASE_PASSWORD --project speqq --config <dev|prod> --plain
doppler secrets get NEXT_PUBLIC_SUPABASE_URL --project speqq --config <dev|prod> --plain
```

Confirmed Supabase-related Doppler keys (from `docker/db.sh`, `scripts/db/repair-migration-history.mjs`, `.claude/SYSTEM.md`):

- `DATABASE_PASSWORD` — Postgres password for `postgres` user on the linked project.
- `NEXT_PUBLIC_SUPABASE_URL` — public Supabase URL; project ref is the leftmost hostname label.
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` — anon/publishable key.
- `SUPABASE_SECRET_KEY` — service role key. Never reference inside a migration.

## Ordering & conflicts

- Timestamp format: `YYYYMMDDHHMMSS_<name>.sql`, generated by `supabase migration new`.
- Lex order = chronological order.
- Applied versions tracked in `supabase_migrations.schema_migrations`.
- Drift between folder and tracking table → use `supabase migration repair` (only sanctioned fix).

## Process loop

1. Branch off `dev` (`<type>/<short-description>`). Never work on `dev` or `main`.
2. `supabase migration new <name>` to scaffold the file.
3. Write SQL. One logical change. No env-specific values, no secrets, no prod data.
4. Apply locally and verify the schema:
   - This repo: ensure local stack is up via `./speqq-local up`, then run the migration through the path documented in `docker/db.sh` (which calls `npx --yes supabase migration up --workdir /workspace --db-url "${LOCAL_DB_URI_DIRECT}"`).
   - Or, on a clean local DB: `supabase db reset --local` to re-apply everything.
5. Run app build and tests against the migrated local DB. Local must pass before anything leaves your machine.
6. Commit the migration in the same commit/PR as the dependent application code. Open PR against `dev`.
7. After merge to `dev`, apply to the dev Supabase project:
   - `doppler run --project speqq --config dev -- supabase db push --linked --dry-run`
   - Review output, then re-run without `--dry-run`.
   - `doppler run --project speqq --config dev -- supabase migration list --linked` to confirm.
8. Verify dev: app smoke tests pass against dev. Dev must pass before prod.
9. Promote to prod (after merge to `main`):
   - `doppler run --project speqq --config prod -- supabase db push --linked --dry-run`
   - Review, then run without `--dry-run`.
   - `doppler run --project speqq --config prod -- supabase migration list --linked` to confirm.
10. If promotion fails partway, do not edit existing migrations. Author a new forward migration that resolves the state, restart at step 2.

## Rules

- Never edit a migration file after it has been pushed to dev or prod. Always write a new forward migration.
- Never run SQL against dev or prod via Studio, psql, or any other direct channel. Migrations only.
- Never hardcode project refs, DB URLs, passwords, or keys. Always Doppler.
- Never backdate or hand-edit migration timestamps. Always use `supabase migration new`.
- One logical change per migration file.
- Migration ships in the same commit/PR as the dependent application code.
- Local must pass before dev. Dev must pass before prod. No skipping environments.
- No secrets, env-specific values, or prod data seeds inside migration files. Seed scripts live elsewhere.
- Destructive migrations (drop column, drop table, type change, adding `NOT NULL` to an existing column, renaming a column referenced by app code) MUST be flagged in the PR description and require explicit reviewer confirmation before promotion to prod. Prefer a multi-step rollout (add new → backfill → switch reads → switch writes → drop old).
- If your branch is behind `dev` and another migration was merged first, rebase, delete your migration, and regenerate it with `supabase migration new` so it gets a fresh timestamp.
- `supabase migration repair` is the only sanctioned tool for drift. Do not mutate `supabase_migrations.schema_migrations` by hand.
- Before declaring done, confirm `supabase migration list --linked` shows zero pending rows in the target environment.
